# 모드별 골격 + 채워진 예시 + 심화 규칙

모든 예시는 2000자 게이트를 통과하는 밀도로 작성됐다. 골격의 섹션은 필요 없는 것부터 지운다 — 골격이 프롬프트를 부풀리는 순간 골격이 틀린 것이다.

## 레인 게이트 카드 — 네거티브 정책 단일 정본

SKILL.md 출력 게이트의 "레인 게이트 카드 통과" 항목은 이 표를 가리킨다. 레인별로 네거티브·비율·길이·필수 요소가 다르므로 상위 게이트 한 줄로 판정하지 않는다.

| 레인 | 네거티브 정책 | 비율/AR | 길이 | 필수 요소 |
|---|---|---|---|---|
| gpt-image-2 | 네거티브 3단 Tier: Tier-0 전부 긍정형(기본) / Tier-1 텍스트 렌더 가드 화이트리스트 / Tier-2 화보 컴플라이언스 페어(명시 선언 시만) — 토큰 세트·승격 조건 정본: [image/compiler.md](image/compiler.md) §2 | 끝 `AR x:y` 토큰 필수 | 블록당 2000자 자기완결 | HEX 3~5(장면 팔레트와 실제 일치), 독립 컷 1행=1컷 |
| Higgsfield (Soul·모션) | 전부 긍정형 | AR 토큰 없음 — `[프리셋/비율 — UI에서 선택]` 라벨 줄로 표기 | 블록당 2000자 자기완결 | 프리셋 1개, Soul HEX는 `Color signature` 레퍼런스 1~20장으로 전역 톤 제어, 클립당 지배 모션 1개, 텍스트 렌더 비의존 |
| COMPOSITE | 연산 금지 계약(crop·resample·beautify 등)은 부정형 정당·필수, 장면 내용은 긍정형 | AR 토큰 불필요 — 프레임 락이 입출력 동일을 강제 | 블록당 2000자 자기완결 — JSON 포맷도 동일, 길이 면제 없음 | 작업 재정의 선언, 재조명 다이얼 1단, 접지 그림자·그레인 통일, 실패 조건 |

| DESIGN 오버레이 | (실행 모드 골격 따름) | — | 블록당 2000자, 오버플로우 문서 허용 | 방향 잠금·시각 테제 3줄·AI 슬롭 금지·뷰포트 검증 |

**단일블록 편집 규칙**: IMAGE·COMPOSITE의 단일 편집 프롬프트는 기본 1개 코드블록으로 합쳐 내보낸다 — 메인 지시와 보존/배제/네거티브성 제약을 별도 블록·메뉴로 쪼개지 말고, 대상 레인 네거티브 정책을 지킨 채 한 번에 붙여넣을 수 있게 통합한다.

**현실감 게이트 (조건부)**: 실사 서브레인(인물·화보·제품·음식·라이프스타일 스틸·배경합성)은 위 필수 요소에 더해 현실감 게이트 — 광원 방향·색온도·카메라/렌즈·불완전성 ≥1·안티-AI 긍정형 앵커 — 를 통과한다. 실사 영상은 시간 일관성·모션 블러 물리성·접지 물리 증분을 추가한다. 정본·면제 목록: [image/realism.md](image/realism.md)(§3 스틸 게이트·§6 영상 증분). 비실사 레인(타이포·아이콘·만화·인포그래픽·카드뉴스·덱)은 면제.

## 슬롯 자동 채움 — 빈칸은 묻기 전에 채운다

짧고 불완전한 요청이 정상 입력이다. 추론 우선순위: ① 대화 컨텍스트(직전 대화·첨부·언급된 프로젝트) → ② 저장소/파일 증거(경로가 언급되면 실제로 읽어 테스트 러너·언어·규모 확인) → ③ 아래 표의 관례 저위험 기본값 → ④ 그래도 결과·하드라인이 바뀌는 빈칸만 질문 ≤3. 채운 값은 전부 `가정:` 노트에 — 틀렸으면 사용자가 한 줄로 고친다. **예외**: 작업 결과 자체(무엇을 만들지)가 요청에 없으면 슬롯 채움 문제가 아니다 — Tie-break ⑥(요구 구체화 인터뷰 권고 또는 질문) 경로다.

| 모드 | 필수 슬롯 | 비었을 때 채움 규칙 |
|---|---|---|
| GOAL | 대상(repo/경로), 완료 증거, 게이트, 스코프 | 대상: 대화에서 마지막 언급 프로젝트, 그것도 없으면 "실행자가 행동 전 대상을 식별·보고"를 계약에 삽입. 완료 증거: 해당 작업의 테스트/명령 출력. 게이트: 필요성 테스트 통과분만(없으면 0개) |
| TEAM | 목표, 범위+비목표, 레인, 검증 | 목표·범위: 대화 컨텍스트에서 — 결과 자체가 없으면 위 예외 경로. 레인: 파일/모듈 경계 기준 2~3개 자동 분할. 검증: 기존 테스트 스위트·측정 스크립트가 있으면 그것을 지정 |
| CONTRACT | 결과, 완료 기준, 검증 | 결과·완료 기준: 대화 컨텍스트+산출물 유형 관례(보고서→섹션 구조 완성, 스크립트→실행 성공)에서 도출. 검증: 산출물을 실제로 열거나 실행하는 최소 명령 1개 |
| BUSINESS | 직무+전문성, 현실 제약, 형식 | 전문성: 경력 10년 내외 국내 실무 기준. 제약: 국내 일반 실무 기준까지만 기본값 — 특정 법령·회계기준·교육과정 버전이 결과를 가르면 가정 노트에 버전 명시 또는 질문 분기. 형식: 산출물 유형 관례(검토서→표+수정안, 수업안→단계표). 난이도: 3~5 |
| MODEL | 대상 모델, 출력 계약 | 모델 미상: 포터블 구조(역할·목표·제약·컨텍스트·형식·검증)로 작성 |
| IMAGE | 피사체, 컷타입, 비율, 팔레트, 카피 | 피사체: 요청 명사·소재에서. 컷타입: 요청 명사 매핑(포스터→구조물, 화보→인물, 아이콘→오브젝트). 비율: 용도 관례(포스터 2:3, 배너·덱 16:9, SNS 1:1). 팔레트: 소재·무드에서 도출. 카피: 렌더 텍스트가 필요한 컷이면 추론 불가(질문 후보), 텍스트 없는 컷이면 슬롯 생략 |
| COMPOSITE | 원본 이미지, 타깃 배경, 재조명 다이얼 | 다이얼 미지정: B(부분 허용). 타깃 배경 무드 미지정: 원본 조명과 어울리는 로케이션 1개 가정 |
| DESIGN | 방향, 시그니처, 하드 제약 | 방향: 제품 도메인에서 저위험 1개를 가정으로 선언(개발자 도구→raw terminal, 커머스→클린 에디토리얼 등). 하드 제약: 저장소 증거(프레임워크)에서 |

**추론 불가 슬롯 — 질문이 정당한 유일 목록 (정본)**: 퇴고·델타의 기준 원문(원문 없는 "다듬어줘"/"톤만 바꿔"), 합성할 원본 이미지, 렌더될 정확한 한글 카피·브랜드명, 실계정·실서비스 식별자, 결과를 가르는 양자택일 방향(둘 다 저위험이 아닐 때). 이 목록 밖의 빈칸으로 질문하면 **질문 반사** 안티패턴이다. SKILL.md의 축약 목록은 이 정본을 가리킨다.

드라이런 캘리브레이션 3종 (이 표가 맞게 작동하는 기준):

1. "flaky 테스트 고치는 goal 하나 써줘" → GOAL. 대상 repo는 대화 컨텍스트에서, 없으면 식별-먼저 계약 삽입. **질문 0** — 바로 작성.
2. "카페 신메뉴 포스터 프롬프트" → IMAGE(gpt-image-2, 구조물). 비율 2:3·팔레트·레이아웃은 기본값, 렌더될 정확한 메뉴명 카피만 질문 후보. **질문 ≤1**.
3. "이 프롬프트 다듬어줘" (원문 첨부 없음) → MODEL 퇴고. 원문은 추론 불가 목록 → **질문 1이 정당**.

## 수신자·채널 적응 — 기본 전제 밖의 실행자

기본 전제는 **툴 사용 가능한 자율 에이전트 + 메신저 1메시지 전달**이다. 요청에 다른 신호("ChatGPT에 붙일", "API 시스템 프롬프트로", "외주한테 줄", "이메일로 보낼")가 있으면 아래를 적용한다. 수신자·채널이 미지정이면 기본 전제 — 가정 노트에 적지 않아도 된다(기본값이므로).

| 수신자 | 문장 수준 적응 |
|---|---|
| 툴 사용 가능 자율 에이전트 (기본) | 툴·파일·명령 사용 가능 전제. 검증 계약 = 실행할 명령. 환경 승인 훅 존재 전제(게이트 필요성 테스트 ②가 대부분 걸러줌) |
| 타 LLM 챗 UI (파일·툴 접근 없음) | 경로 참조 전면 금지 — 필요한 자료는 라벨 블록으로 인라인(컨텍스트 경제의 명시 예외, 단 블록당 2000자는 유지·초과분은 후속 메시지로 분할 선언). 검증 계약은 "응답 안 자가 점검"(예: 출력 전 스키마 대조 1회)으로 변환. 게이트 판정: 훅이 없으므로 테스트 ②는 대체로 통과하지만, 파일·툴 없는 챗이 실제로 할 수 없는 행동의 금지문은 테스트 ①(실행자에게 열려 있는가)에서 탈락 — 노이즈 |
| API 시스템 프롬프트 (무인 상시 규범) | CONTRACT 시스템 프롬프트 변환 적용 + 세션 기억 없음 전제(매 호출 자기완결), 사용자 입력이 들어올 위치·형식을 명시, 에스컬레이션은 "불확실 시 응답 행동"으로 |
| 사람 팀원·외주 | 위임 계약 6요소 내용은 유지, 표현만 전환 — 헤더는 사람용 제목·불릿으로 바꾸고 시스템용 메타어(goal 명령어·출력 게이트·모드명 등)는 제거. 마감·전달 채널·질문 창구 슬롯 추가. 모호어 0 규칙 동일 — 사람도 "적당히"로는 못 일한다 |

| 채널 | 포장 적응 (길이 계약은 채널 불문 블록당 2000자 유지) |
|---|---|
| 텔레그램 (기본) | 블록당 1메시지, N블록=N메시지 |
| 이메일 | 제목 = 결과 1줄, 본문에 프롬프트 블록 그대로, 상세는 첨부 대신 본문 하단 부록 섹션 |
| 문서·위키 삽입 | 프롬프트 블록을 코드펜스로 격리 — 주변 설명 산문과 섞지 않는다(복붙 시 설명이 딸려오면 실패) |
| 크로스호스트/원격 실행자 | 길이 계약 2항 그대로: 경로 참조 금지, 감량해 자기완결 |

## GOAL — 자율 에이전트 goal 지시문

골격(순서 고정 — 자율 루프의 판정자·리뷰어는 지시문 앞부분과 최종 보고만 본다고 가정하라). 런타임별 호출 접두어는 [adapters.md](adapters.md) 참조:

```text
<결과 — 끝난 상태 1~2문장>

완료 기준:
- <최종 응답에서 판정 가능한 증거 기준 — 경로·명령 출력·개수·링크·스크린샷·정확한 blocker 문구>

게이트:
- <게이트 필요성 테스트 통과분만: 행동 이름 + 대체 행동. 시스템 훅이 이미 막는 행동은 쓰지 않는다>

컨텍스트: <답을 바꾸는 최소 사실·경로·현재 증상>
우선순위: <트레이드오프 1줄 — 예: 근본 원인 > 빠른 통과, 기존 패턴 > 새 추상화>
최종 보고에 반드시: 변경한 것 / 검증한 것(명령+결과) / 안 한 것과 이유.
막히면: <기준> → 시도한 것/막힌 지점/필요한 것 형식으로 멈추고 보고.
```

채워진 예시 (488자 실측):

```text
~/src/acme-api 의 flaky 테스트 test_billing_retry 가 100회 연속 통과하는 상태로 만든다.

완료 기준:
- 원인을 코드 라인 단위로 지목 (파일:라인)
- 수정 후 `pytest tests/test_billing.py -k retry --count 100` 출력에 100 passed
- 프로덕션 로직 변경 시 변경 diff 요약 5줄 이내

게이트:
- 테스트 skip/xfail 처리 금지 (대신 원인 수정)
- 변경은 diff로 최종 보고에 포함 (커밋은 이 작업 범위 밖 — 범위 정의)

컨텍스트: 실패는 CI에서만 간헐 재현, 로컬 재현법 미상. tox 환경 py311.
우선순위: 근본 원인 > 빠른 통과.
최종 보고에 반드시: 변경 파일 목록, 100회 실행 출력 요지, 안 한 것과 이유.
막히면: 동일 오류 3회 반복 또는 재현 불가 확정 시 → 시도/막힌 지점/필요한 것 보고 후 종료 (blocked도 유효 종료).
```

### 스코프 분할 규칙 — 턴 예산·판정 명료성이 흔들리면 쪼갠다

| 상황 | 분할 방식 |
|---|---|
| 리서치+구현 혼합 (접근법·영향 파일·수용 테스트 미상) | 리서치 goal 먼저 — DoD = 간결한 계획+정확한 파일+위험+검증 명령. 구현은 후속 goal |
| 광역 리팩토링 (repo 전역·다수 파일) | 서브시스템·API 경계·테스트 표면 단위 분할. 1차 goal은 참조 인벤토리+안전한 순서 제안, 후속 goal이 유한 슬라이스 실행 |
| 승인 게이트 걸린 행동 | 정확한 승인 요청문+증거 준비까지가 goal. 승인 후 후속 goal로 계속 |
| 장기 실행·모니터링 | 무한 goal 금지 — 관찰 시간 박스, 모니터 생성/확인, goal을 끝내는 이벤트 정의, 최종 상태+모니터/잡 ID 요구 |
| 독립 결과 2개+ | 복합 프롬프트 1개가 아니라 별도 goal N개, `분리 제안:`에 명시 |

후속 기준 추가는 별도 goal로 — 한 goal에 미래 요구를 전부 욱여넣지 않는다.

### Higgsfield 레인 — Soul 스틸·Soul ID·카메라 모션 영상 (2026-07 공식 가이드 실측 기준)

- **1장면 = 3프롬프트 잡 분리.** 프레이밍(이미지)·아이덴티티(인물)·모션(영상)을 한 블록에 섞으면 프레임이 불안정해진다. 툴마다 프롬프트를 따로 쓴다 — 이미지 툴엔 시각 레이아웃만, 영상 툴엔 모션·시간만.
- **프리셋 > 일반 프롬프트.** Soul 2.0은 큐레이션 프리셋(Warm Ambient·Flash Editorial·Theatrical Light·Subtle Flash 등 20+)이 일반 서술보다 정제된 결과를 낸다. 무드 요청은 프리셋 1개 지정에서 시작하고, 프롬프트 본문은 피사체·장면·조명 보강만.
- **Soul 스틸 프롬프트 형태.** 6섹션 장문이 아니라 1문단: 피사체(성인 가상 페르소나, 헤어·의상·행동까지 구체적으로) → 장면 → 스타일/무드 → 조명. 전역 색감은 프롬프트 HEX 목록보다 Soul 2.0 웹 `Color signature`에 레퍼런스 1~20장을 넣어 제어한다. 한국 로컬리티는 공간·광고/잡지 톤·의상 소재처럼 보이는 단서로만 작성한다. 텍스트 렌더 정확도는 기대하지 않는다 — 카피가 필요하면 gpt-image-2 컷으로 분리.
- **Soul HEX / 일반 HEX 계약.** 공식 Soul HEX는 레퍼런스에서 팔레트를 추출해 전체 색 구성·무드·컬러 그레이드를 적용하는 기능이다. 대상별 `재질 역할 + #RRGGBB` 하드락이나 수동 3~5 HEX 입력을 Soul의 기본으로 강제하지 않는다. 원본 Color signature 레퍼런스가 권위이며, 타 모델/QC 핸드오프가 필요할 때만 같은 레퍼런스나 승인 결과에서 3~5개 `observed_palette`를 추출해 전역 온도·명암·하이라이트/그림자·액센트 관계의 soft metadata로 쓴다. 피부에는 HEX를 부여하지 않는다. 정확한 의상·제품·브랜드색은 별도 스와치/제품 레퍼런스와 `critical_colors`로 분리하고 결과 QC를 붙인다. image-to-video는 입력 이미지가 색 권위다. 필름명은 색응답·하이라이트 롤오프·질감만 정의하고 필름 테두리/스캔 용기 어휘를 프롬프트에 호출하지 않는다.
- **카메라 모션 영상.** 프리셋 라이브러리(dolly in/out·crash zoom·FPV drone·bullet time·orbit·static 등 100+)에서 **클립당 지배 모션 1개 + 핵심 행동 1개**만 지정. Camera Control이 피사체를 잠그므로 프롬프트는 무브 중에 일어나는 행동만 기술한다. image-to-video는 입력 이미지를 다시 묘사하지 말고 변화·동작·오디오만 쓴다. 클립은 ~10s 단위 — 긴 서사는 스토리보드로 씬 분할 후 씬당 1클립·1모션.
- **Soul ID (동일 인물 시리즈).** 프롬프트 기술이 아니라 사전 훈련 절차: 고품질·다각도·일관 조명 사진 20장+ 업로드(선글라스·짙은 그림자·크롭 얼굴 제외). 훈련 후엔 프리셋·조명·앵글이 바뀌어도 정체성이 유지된다 — 프롬프트에서 외모 재서술로 정체성을 고정하려 들지 말고, 프롬프트엔 장면·스타일·행동만 쓴다.
- **멀티 엔진 허브 주의.** Higgsfield 안에서 타사 영상 엔진(Sora·Kling·Veo·Seedance·Wan 계열)도 호출된다 — 모션은 프리셋 UI가 흡수하므로 프롬프트는 행동·분위기·사운드만. 엔진별 고유 문법을 지어내지 않는다. 본문에 남는 무드/룩 단어도 병기 원칙 적용 — 프리셋 라벨을 본문에 복창하지 말고, 화면에 보일 결과(빛·질감·분위기)로 쓴다.
- 자기완결·2000자·긍정형 재서술·natural skin texture·실재 인물/상표 금지 등 공통 철칙은 동일 적용. 프리셋·모션 지정은 프롬프트 본문이 아니라 UI 선택이므로, 핸드오프 산출물에는 `[프리셋: X — UI에서 선택]` 라벨 줄로 분리해 표기한다.

Soul 스틸 채워진 예시 (측정값은 lint 기준, 자기완결):

```text
[프리셋: Flash Editorial · 비율 2:3 | Color signature: cool rainy-night fashion reference 1장 — UI에서 선택/업로드]
late-20s Korean woman, 짧은 웨이브 단발, 무광 블랙 레더 재킷에 실버 이어커프, 밤의 도심 주차장 콘크리트 기둥 앞에 단독으로 서서 카메라를 정면으로 응시. direct on-camera flash 특유의 강한 정면광, 뒤로 짙게 떨어지는 그림자, 배경은 어두운 콘크리트 톤으로 정리. natural skin texture, visible pores, subtle film grain.
```

카메라 모션 채워진 예시 (image-to-video, 위 스틸 입력, 224자 실측):

### 배경 합성 레인 (COMPOSITE) — 피사체 보존 + 배경 교체

원본 사진(인물·제품)을 그대로 두고 배경만 새로 생성해 합성하는 i2i 편집 작업. 생성 레인과 문법이 다르다 — 실측 검증된 관용구·톤 세팅을 쓴다.

**톤 세팅 3장치** (이 레인 특유의 문체):

1. **작업 재정의 선언으로 시작.** 모델을 "창작 모드"에서 빼낸다: "이 작업은 이미지 생성이 아니라 픽셀 고정 합성(pixel-bound composite)이다. 원본은 고정 사진 원판(locked photographic plate)이며 최종 캔버스를 정의한다." 피사체 레이어는 read-only, 생성 허용 영역은 배경 픽셀뿐임을 첫 문단에 박는다.
2. **판정 문장(judgment sentence)으로 의도 고정.** 형용사 대신 참/거짓 판정문: "조명이 의도된 것처럼 보이면 틀린 것이고, 우연처럼 보이면 맞는 것이다", "동일인으로 즉시 인식되는 것이 미적 개선보다 우선한다", "배경은 조연이고 피사체가 절대다". 끝의 FINAL INTENT 단락에 1~2개.
3. **실패 조건 명시 + FAIL 지시.** 품질 기준을 실패 상태로 전환: "다음 중 하나라도 발생하면 작업 실패 — 피사체 픽셀 이동, 크롭/캔버스 변경, 얼굴 재드로잉, 직물 패턴 변형, 원근 불일치, 떠 있는 발". 캔버스 유지가 불가능하면 어설프게 타협하지 말고 실패로 처리하라고 지시.

**네거티브 정책 (생성 레인과 다름).** 이 레인의 금지문은 장면 네거티브가 아니라 **연산 금지 계약**(crop·resample·beautify·recenter 같은 동작의 이름)이므로 부정형이 정당하고 필수다. 단 **장면 내용**은 여전히 긍정형 재서술: "no crowds" 대신 "행인은 모션 블러·고스트 실루엣으로만, 식별 가능한 얼굴 없음", "no signage" 대신 "간판은 추상적 빛 형태로만, 읽히는 텍스트 없음". 단일 편집 요청의 블록 통합은 §레인 게이트 카드의 단일블록 편집 규칙을 따른다.

**8블록 해부** (샘플 실측 어휘 — 필요한 블록만, 이 순서로):

| # | 블록 | 캐노니컬 어휘 |
|---|---|---|
| 1 | 작업 재정의 | pixel-bound composite, locked photographic plate, subject pixels read-only |
| 2 | 프레임 락 | 출력 크기·비율 입력과 동일, 크롭·줌·팬·리프레이밍·재중앙·여백 재배분 금지, 좌표 1:1, 배경이 프레임을 넘으면 잘라라(리프레이밍으로 배경을 살리지 말 것) |

**재조명 다이얼 — 반드시 한 단을 명시** (샘플들이 갈리는 지점, 미지정이 최다 실패 원인):

| 단 | 계약 | 쓸 때 |
|---|---|---|
| A 무접촉 | 조명은 배경 정의용으로만, 피사체 재조명·에지 강화 일절 금지 | 원본 조명이 이미 목표 무드일 때 |
| B 부분 허용 | 수치 허용치로 조임: 노출 ±0.3 stop 이내, 환경 틴트 ≤0.2, 색온도 이동 ≤200K, 피부 언더톤 보존 ≥0.75 | 대부분의 합성 — 톤만 배경에 동화 |

**원근 정합 2방식** — 배경을 새로 생성할 때: (a) 카메라 지오메트리 명시(높이 m·초점거리 mm·pitch/yaw/roll도, 소실점 일치, 수직선 유지, 어안 금지) 또는 (b) 수동 정합("카메라 파라미터를 추론하지 말고 원판에 시각적으로만 맞춰라"). 하나만 골라 명시 — 둘 다 쓰면 충돌.

**압축 규칙.** 샘플식 불리언 30줄 스택(no_virtual_zoom/pan/shift/dolly...)은 중복 — 캐노니컬 문장 1줄("카메라 가상 이동 일절 금지")로 병합해 2000자 자기완결을 지킨다. JSON 스펙 포맷을 명시 요청받아도 **2000자 이내에서만** — 이미지 계열엔 길이 면제가 없다(넘치면 캐노니컬 압축이 답). SD 폐기 어휘(8K·masterpiece) 금지는 이 레인에도 동일 적용. 엔진 주의: COMPOSITE 문법은 i2i 편집 모델 전제 — 사용자가 특정 엔진(예: Higgsfield)을 지명하면 그 엔진이 픽셀 고정 합성을 지원하는지 확인이 필요하다는 노트를 산출물에 붙인다.

채워진 예시 (인물 스냅 → 성수동 골목, 다이얼 B, 1208자 실측):

## DESIGN 오버레이 — UI/프론트엔드 제작 프롬프트 (관점 있는 디자인 계약)

실행 모드(GOAL/TEAM/CONTRACT) 골격 위에 겹친다. 원리: 디자인 지시는 기능 명세만 주면 실행자가 제네릭 디폴트로 수렴한다 — 취향을 계약으로 강제해야 "관점 있는" 결과가 나온다.

작성 절차:

1. **방향부터 잠근다 — 추론 먼저.** 대화·저장소·제품 도메인에서 사용 맥락(누가 어떤 상황에서 쓰나)과 미학 방향을 먼저 추론해 가정으로 선언한다. 방향은 정확한 이름으로 — dense editorial, raw terminal, ink-on-paper, brutalist grid, warm analog. "클린하고 모던"·"예쁘게"는 방향이 아니다(모호어 치환 대상). 질문은 후보 방향이 둘 다 저위험이 아닌 양자택일일 때만(≤3, §슬롯 자동 채움 추론 불가 목록).
2. **레퍼런스는 분해해서 넣는다.** "유명 SaaS/AI 제품 느낌"을 그대로 프롬프트에 쓰지 않는다 — 구체 속성 3개로 분해: 버튼 radius 철학 / 표면 깊이 처리(그림자 vs 배경 명도 단차 vs 보더) / 액센트 컬러 계열. 분해된 속성만 프롬프트에.
3. **주입 블록을 프롬프트에 박는다** (아래 블록을 실행 모드 골격의 "완료 기준"·"컨텍스트" 사이에 삽입, 과제에 맞게 채움):

## MODEL — 퇴고·델타·리서치/추출·체인

수정본 코드블록 + **문구를 실제로 바꾼 노트만**(에세이 비평 금지). 3렌즈 순서:

1. **프롬프트 아키텍트** — 구조, 컨텍스트 경제, 배치, 모델 적합.
2. **도메인 전문가** — 정확성, 누락 제약, 용어.
3. **UX 라이터** — 명료성, 모호성, 실행 가능성.

두 출력 모드: 신규 작성(붙여넣기 완성본) / 퇴고(수정본+변경 노트). 로컬 파일 프롬프트는 읽고, URL 뒤 원자료는 추출 후 작성 — 프롬프트가 근거할 사실을 지어내지 않는다.

### 델타 수정 — "톤만 바꿔", "이 줄만"

직전 산출 프롬프트의 부분 수정 요청은 퇴고가 아니라 델타 작업이다: **전체 재작성 금지.** ① 변경 축을 1문장으로 복명("톤을 격식→친근으로만 변경") ② 지정 축 밖은 바이트 단위 보존 — 개선이 보여도 손대지 않는다(원하면 별도 제안 노트로) ③ 산출은 수정본 전문 + 바뀐 줄만 나열한 변경 노트 ④ 길이 라벨 재실측 ⑤ **기준 원문이 세션에 없으면 추론 불가 슬롯** — 기준 프롬프트를 요청하고, 기억으로 재구성해 산출하지 않는다. 보존 선언이 없는 델타는 이미지 편집 규칙("X만 바꾸고 나머지 유지")의 텍스트판 위반이다.

### 프롬프트 체인 — A의 산출이 B의 입력

다단계는 하나의 프롬프트로 합치지 않는다. ① 단계별 블록 분리, 각 블록은 자기 모드·게이트 카드 적용 ② **인터페이스 계약**: A의 "산출 형식"과 B의 "입력" 섹션이 같은 스키마를 문자 그대로 공유 — A 출력 예시 1개를 B의 입력 예시로 재사용해 정합을 증명 ③ 중간 산출 검증: 각 단계에 "다음 단계로 넘기기 전 확인" 1줄 ④ 블록당 2000자, 체인 전체 개요는 노트에 1줄("체인: 추출→분석→보고서, 블록 3개").

## 보강 루프 — 산출 후 후속 요청 처리 (상세)

SKILL.md §보강 루프의 상세판. 진입점은 세 질문으로 정한다: **① 결과 명세가 보존인가 ② 모드·레인이 보존인가 ③ 계약 요소 몇 개를 건드리나.** 셋 다 보존이면 패치, 하나라도 깨지면 워크플로우 1단부터 재라우팅.

### 유형별 처리

- **(a) 스코프 추가** — 새 요구가 **같은 결과**의 하위면 완료 기준·게이트·자료에 삽입 패치, **독립 결과**면 하나로 합치지 말고 별도 블록/goal 분리(Tie-break ⑦, GOAL §스코프 분할 규칙). 판정: 새 요구가 기존 DoD로 판정되면 패치, 자기 DoD가 필요하면 분리.
- **(b) 방향 수정** — **단일 축**(톤·형식·우선순위 1개)이면 델타(§MODEL 델타 수정) — 지정 축만, 나머지 바이트 보존, 변경 노트. **결과 명세·모드·레인**이 뒤집히면 델타로 때우지 말고 재작성(패치가 남긴 모순이 더 비싸다).
- **(c) 압축/감량** — SKILL.md §길이 계약 감량 순서(장식→중복 병합→how 다이어트→오버플로우/컷 분할) 재적용 후 재실측. **계약 요소(결과·DoD·게이트·검증)는 감량 대상이 아니다** — 요구가 계약을 지울 만큼 짧으면 오버플로우 문서(동일 FS)·이미지 컷 분할로 옮긴다. "짧게"가 계약을 지우는 핑계가 되면 실패.
- **(d) 실행 실패 환류** — 프롬프트대로 했는데 결과가 나빴다. 증상을 계약 결함으로 진단해 **그 요소만** 고친다(전면 재작성 아님). SKILL.md 3층·6요소 표의 역인덱스:

  | 실패 증상 | 결함 요소 | 고침 |
  |---|---|---|
  | "다 했다"는데 안 돼 있음·얕은 산출 | DoD가 가짜 완료 통과 | 가짜 완료 테스트로 증거 요구 조임 |
  | 엉뚱한 걸 만듦 | 결과가 활동(상태 아님) | 결과=끝난 상태로 재서술 |
  | 사고성 행동(지울 것 아닌 걸 지움) | 하드라인·게이트 누락 | 필요성 테스트 통과분 게이트 추가 |
  | 자꾸 멈춰 되물음 | 자율성 3단 불명확 | 혼자 결정 범위 확대 |
  | 막혀서 빈손 종료 | 에스컬레이션 형식 없음 | blocked 형식(시도/막힌 지점/필요한 것) |
  | 검증 없이 보고 | 검증 계약 누락 | 자가 검증 명령 추가 |
  | 형식이 매번 다름 | 구조층 예시 없음 | 채워진 예시 1개 추가 |
  | 특정 모델만 이상·잘림 | 표면층 문법 부적합 | model-playbooks 재적용 |
  | 이미지: 배제한 게 등장·글자 중복 | 레인 게이트 카드 위반 | compiler.md 철칙 역추적 표 |

- **(e) 모호("좀 더 좋게")** — 모호어다. 되묻기 전 전달본을 SKILL.md 3층 설계 모델 표에 대조해 **최약 축**을 진단·개선하고 `가정: '좋게'를 <축> 강화로 해석`으로 투명화. 질문은 후보 축이 둘 이상 대등하고 서로 다른 재작성을 낳을 때만 ≤1(질문 반사 회피). 구체 증상이 오면 (d) 표를 쓴다.

### 재검증 불변식 (보강 후 필수)

보강은 신규 산출과 **동일한 출력 게이트**를 다시 통과한다 — 특히 블록당 2000자 실측, 이미지 자기완결, 레인 게이트 카드, 게이트 필요성 테스트. 패치가 블록을 2000자 넘기면 SKILL.md §길이 계약 그대로 감량 → 그래도 초과면 오버플로우 문서(동일 FS일 때만)·이미지 컷 분할. 델타·압축은 길이 라벨을 재실측한다.

### 체인 무결성

전달본이 멀티블록 체인(A 산출→B 입력)의 일부면, 보강이 **인터페이스(산출 스키마)를 건드리는지** 먼저 판정한다. 건드리면 §MODEL 프롬프트 체인 정합을 재증명 — A의 산출 형식을 바꿨으면 B의 입력 섹션·공유 예시도 같은 스키마로 동반 수정한다(안 하면 체인이 조용히 깨진다). 인터페이스 밖 내부 보강(로직·게이트·문구)은 단독 패치. 블록 수가 바뀌면 체인 개요 노트도 갱신한다.

### 자동개선 옵션 추천 (산출 시 선제 진입점)

보강 루프는 사용자 요청만이 아니라 **산출 직후 자가 진단**으로도 연다. 단 **조건부** — self-critique로 실제 약한 축(검증 조항 얇음 / `가정:` 슬롯 과다 / 감량으로 뉘앙스 손실 / 3층 최약 축)이 잡힐 때만 산출물 뒤에 두 줄을 붙인다. 게이트를 깨끗이 통과하고 채운 슬롯도 저위험이면 **침묵**(상시 형식 추천은 노이즈).

형식 — 산출물 뒤 정확히 2줄:

- `진단: <모드/레인> — <약한 축 1개, 구체적으로>`
- `자동개선 옵션: <약한 축>만 self-critique→재작성 원하면 알려주세요 (결과·모드·레인 그대로)`

예: `진단: GOAL — 검증 조항이 완료 기준 대비 얇음(명령 1개).` / `자동개선 옵션: 검증 축만 조여 재작성할까요? (델타)`

**자동개선 패스 정의**: 진단된 약한 축 대상 self-critique → 재작성 → 동일 출력 게이트 재통과(§재검증 불변식). 수락 시 축 성격대로 흘려보낸다 — 단일 축 조임=(b) 델타, 길이=(c) 감량, 모호·복합 약축=(e) 최약 축 개선. **과개선 방지**: (e)로 가더라도 진단된 축 밖은 보존, 계약 블록(결과·DoD·게이트·검증)은 삭제·압축 금지, 강조어 증식 금지 — 무인 패스라 "개선"이 남용 방향으로 쏠린다.

**경계**: 결과 명세·모드·레인을 뒤집는 변경(방향 수정)은 자동개선 소관 밖이다 — 진단에 그런 신호가 보이면 옵션으로 제안하지 말고 "방향을 바꾸려면 보강요청 필요"로 표기한다. 자동개선은 계약 보존 범위 안의 축만 조인다.

**중단 기준**: 반복 자동개선("계속 좋게")은 라운드 상한이 아니라 **진단이 끝낸다** — 재진단에서 약한 축이 안 잡히면 중단하고 "약한 축 없음, 남은 개선은 방향 변경(보강요청)뿐"으로 알린다(수확체감 스파이럴·과개선 차단).
